To understand how usage of the Inkeep search or chat affects your business, you may want to log events from the search and chat widgets to your own analytics tool. This could be Mixpanel, Posthog, Amplitude, Segment or other analytics or CDP tools.

We expose the events through a callback function called `onEvent` that can be configured in the `baseSettings` configuration.

Below is an example of how to use `onEvent` to log all events to your analytics tool.

## Example
```typescript
import { type InkeepCallbackEvent } from "@inkeep/agents-ui-cloud";

const baseSettings = {
  onEvent: async (event: InkeepCallbackEvent) => {
    // console.log('event', event);
    // log the event to your own analytics tool for example PostHog
    posthog.capture(event.eventName, {
      ...event.properties,
    });
  }
};
```

<Callout type="info">
  Analytics SDKs automatically attach the active user identity at capture time, so Inkeep events land on the same user timeline as your existing events — no extra wiring needed for funnels, cohorts, or user paths.
</Callout>

## Built-in Events API Logging

In addition to the `onEvent` callback, the chat widget automatically logs chat-tree events to the [Events API](/talk-to-your-agents/events) (`POST /run/v1/events`). The following events are sent:

- `user_message_submitted`
- `assistant_message_received`
- `assistant_answer_displayed`
- `assistant_message_copied`
- `assistant_source_item_clicked`
- `assistant_code_block_copied`
- `assistant_message_inline_link_opened`
- `chat_error`
- `user_escalation_indicated`
- `assistant_positive_feedback_submitted`
- `assistant_negative_feedback_submitted`
- `get_help_option_clicked`

Each event includes the `conversationId`, `messageId`, `userProperties` (from `baseSettings.userProperties`), a `properties` bag (containing `baseSettings.tags` and `baseSettings.analyticsProperties`), and a `metadata` bag with widget-specific context like `widgetLibraryVersion`, `componentType`, and event-specific payload.

Search-side events (`search_query_*`, `search_result_clicked`) and widget trigger events (`modal_opened`, `chat_bubble_opened`, etc.) only fire the `onEvent` callback — they are **not** sent to the Events API.

### Privacy opt-out

When `baseSettings.privacyPreferences.optOutAllAnalytics` is `true`, the widget skips the Events API call but `onEvent` still fires. This lets you respect user privacy preferences while keeping your own client-side analytics operational.

### Custom events endpoint

Use `aiChatSettings.analyticsApiBaseUrl` to route event requests to a separate base URL. When omitted, events are sent to `aiChatSettings.baseUrl`.

```typescript
const aiChatSettings = {
  baseUrl: "https://your-api.example.com",
  appId: "YOUR_APP_ID",
  analyticsApiBaseUrl: "https://your-analytics-api.example.com", // optional
};
```
